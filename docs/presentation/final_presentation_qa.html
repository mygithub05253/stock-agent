PoCaT BDAI 12기 1조 · 5분 Q&A 대비

stock-agent 예상 질문과 답변

답변은 20~35초를 기본으로 준비했습니다. 질문이 깊어지면 “현재 MVP 범위”와 “다음 개선 방향”을 분리해서 답하세요.

답변 구조결론 1문장 → 근거 2개 → 한계/다음 단계 1문장.
금융 표현매수·매도 추천이 아니라 분석 보조, 근거 정리, 리스크 경고로 말합니다.
모르는 질문추측하지 말고 MVP 범위 밖이라고 인정한 뒤 확장 방향을 말합니다.
시간 운영5분이면 4~6문항입니다. 한 답변이 1분을 넘지 않게 자릅니다.
Q1

왜 단일 LLM이 아니라 멀티 Agent 구조를 썼나요?

높음

짧은 답변

주식 분석은 계산, 검색, 비교, 거시 해석, 안전 검증이 섞여 있어서 하나의 LLM에 맡기면 오류 위치를 추적하기 어렵습니다. 그래서 책임을 나눠 설명 가능성과 fallback 안정성을 높였습니다.

보충

  • Quant는 Tool/DB 기반 수치 계산, Qual은 뉴스 RAG evidence, Competitor는 peer 비교, Macro는 시장 환경을 담당합니다.
  • 한 Agent가 실패해도 전체 답변이 멈추지 않고 warning/fallback으로 Strategist가 보수적으로 합성합니다.

피해야 할 표현

“멀티 Agent라서 무조건 더 정확합니다” 대신 “책임 분리와 검증 가능성이 좋아집니다”라고 답합니다.

Q2

기존 증권사 MTS나 로보어드바이저와 가장 큰 차이는 무엇인가요?

높음

짧은 답변

MTS는 정보를 많이 보여주지만 사용자가 직접 종합해야 하고, 로보어드바이저는 자산배분 중심입니다. stock-agent는 사용자의 보유 종목과 투자성향을 컨텍스트로 넣고, 개별 종목 근거를 여러 Agent 관점에서 합성합니다.

보충

  • 같은 종목이라도 이미 비중이 높은 사용자와 신규 검토 사용자의 리스크 설명이 달라질 수 있습니다.
  • 추천 엔진이 아니라 “내 포트폴리오 기준으로 근거를 정리해주는 PB-lite 분석 보조”에 가깝습니다.
Q3

GLM을 선택한 이유는 무엇인가요?

중간

짧은 답변

이번 MVP에서 GLM은 핵심 판단 엔진이라기보다 최종 설명과 narrative 보완에 쓰는 LLM provider입니다. 비용, 응답 속도, 한국어 설명 품질, API 연동 편의성을 고려했고, 구조적으로는 다른 LLM으로 교체 가능하게 분리했습니다.

보충

  • 정량 수치와 데이터 검색은 LLM이 아니라 Tool/DB/RAG가 담당합니다.
  • LLM provider가 바뀌어도 Agent 인터페이스와 Guardrail 흐름은 유지되도록 설계했습니다.
  • 질문이 더 깊어지면 “모델 비교 평가는 다음 단계이며, 현재는 provider abstraction과 fallback 가능성을 더 중요하게 봤다”고 답하면 됩니다.

피해야 할 표현

“GLM이 가장 성능이 좋아서 선택했습니다”처럼 검증하지 않은 절대 비교는 피합니다.

Q4

투자 권유나 금융 규제 리스크는 어떻게 피하나요?

높음

짧은 답변

서비스 포지션을 자동 매매나 확정 추천이 아니라 분석 보조로 제한했고, Guardrail에서 수익 보장, 직접 매수·매도 권유, 과도한 확신 표현을 점검합니다.

보충

  • 결과는 BUY/SELL 명령이 아니라 HOLD/주의/근거 카드/리스크 경고 중심입니다.
  • 민감정보나 투자 권유성 표현은 차단하거나 완화하고, 근거 부족 시 warning으로 승격합니다.
  • 제품화 단계에서는 고지 문구, 감사 로그, 금융 규제 검토가 추가되어야 합니다.
Q5

데이터가 틀리거나 누락되면 잘못된 답을 내지 않나요?

높음

짧은 답변

그 위험을 줄이기 위해 데이터 실패를 숨기지 않고 fallback_reason과 warning으로 남깁니다. Strategist는 그런 신호를 받아 보수적으로 합성하고, Guardrail은 근거 부족을 다시 점검합니다.

보충

  • Quant는 DB/API 실패 시 보수적 fallback을 남깁니다.
  • Qual은 RAG 검색 실패 시 출처 없는 요약을 만들기보다 fallback evidence와 reason을 전달합니다.
  • 다음 단계는 데이터 freshness, cache 이력, citation span 품질을 더 강화하는 것입니다.
Q6

RAG, Hybrid/RRF, reranker를 왜 같이 썼나요?

중간

짧은 답변

뉴스와 공시는 키워드가 정확히 맞는 경우도 있고 의미적으로 비슷한 문서를 찾아야 하는 경우도 있습니다. 그래서 의미 검색과 키워드 검색을 같이 쓰고, RRF로 순위를 합친 뒤 필요하면 reranker로 후보를 다시 정렬합니다.

보충

  • pgvector는 의미 검색, ts_rank는 키워드 기반 검색에 가깝습니다.
  • RRF는 두 검색 결과의 순위를 안정적으로 합치기 위한 방법입니다.
  • reranker는 top-k 후보의 최종 관련도를 높이는 선택적 단계입니다.
Q7

정확도나 성능 검증은 어떻게 했나요?

중간

짧은 답변

이번 발표에서는 검증 수치를 전면에 내세우기보다 구조와 흐름을 중심으로 설명했습니다. 다만 Agent별 rule check, RAG 검색 hit, fallback 동작처럼 기능 단위 검증은 진행했고, 정량적 벤치마크는 다음 단계로 남겼습니다.

보충

  • 지금 MVP에서 중요한 검증은 “환각을 줄이는 구조가 있는가”, “실패가 표시되는가”, “출력 전에 안전 검사가 있는가”입니다.
  • 제품화 전에는 종목별 gold set, 전문가 평가, retrieval/citation 품질 평가가 필요합니다.

피해야 할 표현

수익률 개선을 검증했다고 말하지 않습니다. 현재는 투자 성과 예측 검증이 아니라 분석 근거 구조화 MVP입니다.

Q8

시연을 라이브 대신 영상으로 한 이유는 무엇인가요?

높음

짧은 답변

Agent 시스템은 DB, 외부 API, LLM 응답에 영향을 받기 때문에 발표장에서 네트워크나 응답 지연 리스크가 있습니다. 그래서 실제 구현 화면을 영상으로 준비해 같은 사용자 흐름을 안정적으로 보여주려 했습니다.

보충

  • 영상은 온보딩, 포트폴리오 입력, Agent 진행, 결과 확인을 모두 포함합니다.
  • 라이브가 아니어도 실제 UI와 결과 화면 기반이라 구현 흐름 설명에는 충분합니다.
Q9

상업화한다면 누가 돈을 내고 쓸까요?

중간

짧은 답변

개인투자자용 PB-lite 리포트, 증권·핀테크 앱의 설명형 AI 분석 모듈, 투자 교육용 튜터가 현실적인 방향입니다. 핵심은 매매 자동화가 아니라 보유 종목 점검과 근거 설명입니다.

보충

  • B2C는 정기 리포트, 포트폴리오 모니터링, 프리미엄 분석으로 확장 가능합니다.
  • B2B는 기존 MTS에 설명형 AI 모듈로 붙을 수 있습니다.
  • 교육 시장에서는 투자 판단 과정을 시각화하는 튜터로 활용할 수 있습니다.
Q10

가장 큰 한계는 무엇이고 다음에 무엇을 개선해야 하나요?

높음

짧은 답변

가장 큰 한계는 데이터 품질과 커버리지입니다. 다음 단계는 RAG citation span, context gating, reranker 운영 기준, 사용자/보유 종목/분석 이력의 영속화를 강화하는 것입니다.

보충

  • 현재는 MVP라 종목 범위와 수집 안정성이 제품 수준까지 완성됐다고 보기는 어렵습니다.
  • 분석 이력이 쌓이면 사용자별 리스크 변화와 알림 기능까지 확장할 수 있습니다.
  • 운영비 측면에서는 캐시, 배치 수집, 모델 라우팅 최적화가 필요합니다.
Q11

개인정보나 포트폴리오 정보는 어떻게 보호하나요?

중간

짧은 답변

현재 발표 MVP에서는 실제 민감 개인정보를 저장하지 않는 방향으로 설계했고, Guardrail에서 PII 노출을 검사합니다. 제품화 단계에서는 암호화, 접근권한, 감사 로그, 데이터 삭제 정책이 필요합니다.

보충

  • 포트폴리오 데이터는 분석 컨텍스트로 쓰이지만 외부에 노출되어서는 안 되는 정보입니다.
  • 따라서 provider 호출 시 최소 정보만 넘기고, 저장 데이터와 로그 정책을 분리해야 합니다.
Q12

팀원별 기여는 어떻게 확인할 수 있나요?

낮음

짧은 답변

발표 2번 슬라이드에 역할을 앞쪽으로 배치했고, 각 Agent 설명도 해당 구현 책임과 연결했습니다. PR과 README 기준으로 PM/설계, UI/스트리밍, Quant/Guardrail, Macro/DB, 뉴스 RAG 역할을 나눠 확인했습니다.

보충

  • 질문이 들어오면 “역할을 단순 파트가 아니라 Agent 책임 단위로 연결했다”는 점을 강조합니다.
  • Qual RAG처럼 팀원이 별도로 준비한 자료는 해당 Agent 설명 슬라이드에 반영했습니다.

마지막 10초 답변 템플릿

“저희 프로젝트의 핵심은 투자 판단을 대신하는 것이 아니라, 개인투자자가 판단에 필요한 근거를 Agent별로 분해하고 다시 합성해서 볼 수 있게 만든 것입니다. 아직 데이터 품질과 제품화 과제가 남아 있지만, 그 한계를 명확히 드러내고 개선 가능한 구조로 설계했습니다.”
