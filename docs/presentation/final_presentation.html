Why Multi-Agent

왜 멀티 Agent인가

06
1

계산과 해석 분리: 정량 수치는 Tool/DB가 만들고 LLM은 설명과 합성에 집중

2

관점 분리: Quant·Qual·Peer·Macro가 서로 다른 실패 범위와 책임을 가짐

3

부분 실패 허용: 한 Agent가 실패해도 fallback과 warning으로 분석 흐름 유지

4

최종 안전 검증: Guardrail이 투자 권유, PII, 근거 부족 표현을 마지막에 점검

Takeaway한 모델이 모든 판단을 말하는 구조보다, 역할을 나눠 계산·검색·합성·검증을 분리하는 구조가 더 설명 가능하다.

Agent 1

Curator·Classifier: 질문을 실행 계획으로 바꾼다

09

사용자 자연어와 포트폴리오를 분석 가능한 종목·섹터·worker_plan으로 구조화하는 앞단 Agent입니다.

01

사용자 질의·보유 포트폴리오 수신

02

분석 대상 종목/후보와 섹터 확정

03

intent·scope·urgency·depth 분류

04

필요 Agent worker_plan 생성

Input user_query, portfolio, onboarding profile
Decision single_stock / sector / portfolio 범위와 macro 실행 여부
Output CuratorResult, RequestClassification, graph_route
TakeawayAgent 구조의 출발점은 '무엇을 분석할지'와 '어떤 Agent를 실행할지'를 먼저 고정하는 것이다.
Agent 2

Quant Agent: LLM이 숫자를 만들지 않게 한다

10

DART 재무와 pykrx 시세를 기반으로 PER/PBR, 성장률, 수익성, 재무 리스크를 계산합니다.

01

Curator가 확정한 stock_code/corp_code 수신

02

시세와 재무제표 Tool 호출

03

PER/PBR/성장률/부채비율 등 지표 계산

04

score, valuation_signal, reasons, risks 반환

원칙 수치 생성은 Tool/DB 담당, LLM은 임의 보정 금지
Fallback DB/API 실패 시 보수적 fallback과 warning 남김
전달값 Strategist가 가중 합성할 정량 score와 핵심 이유
Takeaway정량 Agent는 설명 가능한 계산 근거를 제공해 LLM 환각 위험을 낮춘다.
Agent 3

Qual Agent: 뉴스 수집에서 evidence/risk까지

11

뉴스 수집 → RAG 임베딩 → Hybrid/RRF → reranker → evidence/risk 생성 흐름입니다.

01 뉴스 수집

네이버 금융 종목 뉴스와 공시 문서 적재

02 RAG 임베딩

raw_news → docs/chunks, bge-m3 벡터화

03 Hybrid/RRF

pgvector 의미 검색 + keyword ts_rank를 RRF로 결합

04 Reranker

선택적으로 top-k 후보를 cross-encoder 기준 재정렬

05 QualResult

evidence, sentiment, event_type, risk, fallback_reason 생성

담당 자료 docs/agents/news_rag_pipeline.html, docs/agents/qual_agent_flow.html 기반
핵심 가치 출처 없는 요약이 아니라 검색 근거를 Qual evidence로 전달
안정성 데모에서 뉴스 fallback이 보이면 DB/임베딩 연결 실패 신호로 보고 연결 상태를 점검
TakeawayQual Agent는 '뉴스를 읽어준다'가 아니라, 검색 가능한 근거를 만들어 정성 리스크로 변환한다.
Agent 4

Competitor Agent: 같은 섹터 안에서 상대 위치를 계산한다

12

대상 종목의 peer를 찾고, 밸류에이션·수익성·성장성·재무 안정성을 비교합니다.

01

Curator가 확정한 sector와 stock_code 수신

02

company, stock_price, financial_statement에서 peer 후보 조회

03

시총 band와 복합 유사도로 비교군 선정

04

상대 위치 score, warnings, peer_summary 반환

3단 경로 DB 성공 → MCP 실시간 시세 → 데모 최후 보루 mock
계산 기준 PER/PBR/ROE/성장률/마진/부채비율/데이터 품질
역할 사용자에게 '내 종목이 업종 내 어디쯤인지' 설명
TakeawayPeer 비교를 분리했기 때문에 단일 종목 분석이 업종 맥락 없이 흐르는 문제를 줄인다.
Agent 5

Macro Agent: 개별 종목을 거시 환경 안에 놓는다

13

금리, 환율, CPI, GDP, 뉴스심리지수 등 거시 지표를 업종 영향으로 해석합니다.

01

sector와 as_of_date 기준 macro context 조회

02

금리·물가·성장·환율·심리 지표를 정규화

03

업종별 민감도에 따라 score와 risks 계산

04

portfolio/sector 분석 또는 sector가 있는 단일 종목에서 조건부 실행

조건부 실행 Classifier worker_plan에 macro가 있을 때만 Send
해석 방향 금리/환율/물가가 업종 수익성에 미치는 압력 설명
Fallback macro DB/tool 실패 시 보수적 지표와 warning 반환
TakeawayMacro는 개별 종목 판단이 시장 환경과 분리되어 보이는 문제를 보완한다.
Agent 6

Strategist·Analyst: 근거를 개인 맥락으로 합성한다

14

Worker 결과를 단순 평균하지 않고, 사용자 보유 비중과 투자성향을 반영해 최종 분석 신호를 만듭니다.

01

Worker Agent 결과 join

02

Macro 포함 여부에 따라 가중치 재조정

03

보유 비중·위험성향·투자기간으로 suitability 보정

04

sLLM/GLM이 thesis와 risk narrative를 보완

가중 합성 Quant 40~45%, Qual 25~30%, Competitor 20~25%, Macro 15%
개인화 같은 종목이라도 보유 비중과 위험성향에 따라 suitability가 달라짐
Fallback LLM 실패 시 rule-based Strategist 결과를 유지
Takeaway개인화는 중간 근거를 바꾸는 것이 아니라, 최종 해석과 적합도에서 반영한다.
Agent 7

Guardrail·Renderer: 안전하게 보여주는 마지막 단계

15

최종 출력 전 PII, 욕설, 수익 보장, 투자 권유성 표현, 근거 부족을 점검하고 Tier 결과로 렌더링합니다.

01

Strategist 결과를 Guardrail 7게이트로 검사

02

PII나 금지 표현 발견 시 보수적 HOLD와 마스킹 적용

03

근거 부족은 recomposer revision loop로 최대 2회 재합성

04

ResultRenderer가 Tier 1/2/3와 다운로드 산출물 구조화

Safety 투자 권유·수익 보장·민감정보 노출을 최종 차단
Transparency fallback과 근거 부족은 사용자 경고로 승격
Output 한 줄 신호, 근거 카드, 상세 리포트로 분리
Takeaway좋은 분석보다 중요한 것은 위험한 표현을 그대로 내보내지 않는 구조다.
