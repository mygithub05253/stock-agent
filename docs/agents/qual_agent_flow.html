Qual Agent 정성 분석 흐름

담당: 김도예 · 범위: 뉴스/공시 RAG 검색, fallback 처리, 정성 evidence·risk·sentiment 산출

Agent 목적

Qual Agent는 Curator가 확정한 종목을 기준으로 뉴스와 공시 근거를 검색하고, 해당 근거에서 정성 점수, sentiment, event type, evidence, risk를 만들어 Strategist Agent로 넘깁니다.

핵심 개선점

기존 keyword/빈 검색 흐름을 rag_chunks 기반 Hybrid/RRF 검색으로 연결하고, DB나 임베딩 환경이 불안정해도 fallback evidence를 남기도록 정리했습니다.

1. 처리 흐름

Input

Curator 결과 수신

stock_code, corp_code, 기업명을 AgentState에서 읽습니다.

Health

DB 연결 확인

ensure_database_available()로 외부 의존성 실패를 먼저 감지합니다.

Retrieve

뉴스/공시 검색

retrieve_news, retrieve_disclosures가 RAG chunk를 검색합니다.

RRF

Hybrid Ranking

vector similarity와 keyword rank를 RRF로 합산해 근거 문서를 정렬합니다.

Judge

Risk/Sentiment

위험 키워드와 evidence 개수를 기준으로 risk, sentiment, score를 계산합니다.

Output

QualResult

점수, 감성, 이벤트 타입, 근거, 리스크를 다음 Agent로 전달합니다.

2. 검색 구조

단계 사용 파일 설명
Query Embedding src/stock_agent/rag/retriever.py 사용자/Agent query를 BAAI/bge-m3로 임베딩합니다.
Vector Search rag_chunks.embedding embedding <=> query_embedding 기준으로 의미적으로 가까운 chunk를 찾습니다.
Keyword Search rag_chunks.search_tsv plainto_tsquery, ts_rank_cd로 키워드 일치 근거를 찾습니다.
RRF Fusion RRF_K = 60 vector rank와 keyword rank를 합쳐 검색 결과의 안정성을 높입니다.
Evidence Formatting format_doc_as_evidence() 근거 문장에 method, rrf_score, similarity, keyword_score, URL을 남깁니다.
score
기본 50점에서 sentiment와 evidence/risk 개수로 보정
sentiment
evidence가 risk보다 많으면 positive, 적으면 negative
event_types
실적, 산업 트렌드, M&A, 규제/리스크, 매크로 등 키워드 기반 분류

3. Fallback 설계

  • DB 연결 실패, API 인증 오류, timeout, rate limit 등 외부 의존성 오류는 fallback 대상입니다.
  • 뉴스 검색 실패 시 fallback_news_docs()가 보수적인 뉴스 근거를 생성합니다.
  • 공시 검색 실패 시 fallback_disclosure_docs()가 DART 재확인 필요 문구를 생성합니다.
  • fallback 근거에는 fallback_reason을 남겨 Guardrail이 신뢰도 경고를 낼 수 있습니다.

4. Guardrail 연결

  • Qual evidence에 fallback 문구가 포함되면 Guardrail에서 mock/fallback 의존도를 감지합니다.
  • 최종 투자 의견에는 실제 DB/API 재확인 필요성이 경고로 반영됩니다.
  • 즉, 실패를 숨기지 않고 “분석은 계속 진행하되 신뢰도는 낮춘다”는 구조입니다.

5. 발표 때 말할 핵심

Qual Agent는 뉴스와 공시를 정성 분석 근거로 바꾸는 Agent입니다. 이번 작업으로 뉴스는 rag_documents → rag_chunks.embedding까지 연결됐고, Qual 검색은 vector 검색과 keyword 검색을 RRF로 합쳐 evidence를 가져옵니다. 외부 DB나 임베딩 환경이 실패해도 fallback evidence와 reason을 남기므로 파이프라인은 멈추지 않고 Guardrail까지 신뢰도 정보가 전달됩니다.
