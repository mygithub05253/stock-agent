Macro Agent 거시경제 분석 흐름

범위: ECOS API 데이터 수집, DB 적재, 업종별 거시경제 점수 산출, LangGraph Send API 동적 라우팅

Agent 목적

Macro Agent는 한국은행 ECOS API에서 수집한 거시경제 지표를 기반으로 종목이 속한 업종의 투자 환경을 평가합니다. 금리·인플레이션·경제성장·시장심리 4개 영역을 0~100점으로 점수화하고 업종별 환율·금리 민감도를 가중치로 보정해 MacroResult를 Strategist에 전달합니다.

핵심 개선점

기존에는 단일 종목 질문에서 Macro가 무조건 스킵됐습니다. 업종이 확인된 종목이라면 거시경제 환경이 주가에 직접 영향을 미치므로 _worker_plan 조건을 수정해 single_stock도 Macro 분석에 포함되도록 했습니다. 또한 LangGraph Send API를 적용해 Macro 노드를 진짜 동적 병렬 실행 구조로 전환했습니다.

1. 처리 흐름

Input

Curator 결과 수신

sector(업종명)를 AgentState에서 읽습니다. 업종이 없으면 공통 지표로 분석합니다.

Map

업종명 정규화

"반도체" → 반도체IT, "은행" → 금융 등 intake.py 업종명을 macro_tool 키로 변환합니다.

Fetch

DB 조회

raw_macro에서 업종 관련 지표 최신값과 이전 시점값을 조회합니다. 실패 시 mock fallback으로 대체합니다.

Score

4영역 점수화

금리·CPI·GDP·NSI를 논문 기반 임계값과 비교해 점수를 가감합니다. 업종별 환율·금리 보정치를 추가 적용합니다.

RoC

변화율 보정

금리·CPI 전기 대비 변화율(RoC)을 계산해 추세 방향을 점수에 반영합니다. 주기 불일치 시 생략합니다.

Output

MacroResult

score, signal(BUY/HOLD/SELL), indicators, reasons, risks, sector를 Strategist로 전달합니다.

2. 데이터 수집 구조 (collector.py)

단계파일설명
API 요청 datas/macro/collector.py 한국은행 ECOS API에서 최근 6개월치 데이터를 수집합니다. 지표별 주기(일/월/분기/연)를 자동 변환합니다.
지표 수집 INDICATORS 딕셔너리 기준금리·환율·코스피·CPI·GDP·뉴스심리지수 등 15개 지표를 업종별 매핑과 함께 정의합니다.
DB 저장 raw_macro 테이블 ON CONFLICT DO NOTHING으로 중복 방지. source=ECOS, indicator_code, observed_at, payload(JSON) 구조입니다.
DB 조회 src/stock_agent/tools/macro_tool.py 업종별 SECTOR_INDICATORS 목록으로 관련 지표만 최신값 조회합니다.

5. Fallback 설계

  • DB 연결 실패·timeout 시 mock 데이터(기준금리 3.5%, 환율 1380원 등)로 분석을 계속합니다.
  • fallback 사용 시 reasons에 "DB 연결 실패로 데모용 추정치 사용" 경고를 포함합니다.
  • Guardrail이 fallback 문구를 감지해 최종 출력 신뢰도 경고에 반영합니다.
  • 실패를 숨기지 않고 "분석은 계속하되 신뢰도는 낮춘다"는 구조입니다.

6. LangGraph Send API 적용

  • 기존: _route_workers가 항상 4개 노드 고정 라우팅 → Macro 내부에서 스킵 처리.
  • 변경: _fanout_workers가 worker_plan 기반으로 Send를 동적 발송.
  • Macro가 worker_plan에 없으면 노드 실행 자체가 안 됨 → 불필요한 실행 제거.
  • single_stock도 업종이 확인되면 worker_plan에 macro 포함 → 거시 분석 적용.

7. 발표 때 말할 핵심

Macro Agent는 한국은행 ECOS API 데이터를 업종별 거시경제 투자 신호로 변환하는 Agent입니다. 이번 작업으로 단일 종목 질문에서도 업종이 확인되면 Macro 분석이 포함되고, LangGraph Send API를 통해 Quant·Qual·Competitor와 진짜 병렬로 실행됩니다. DB 연결 실패 시에도 fallback mock 데이터로 파이프라인이 멈추지 않으며, fallback 사용 여부는 reasons에 명시되어 Guardrail까지 신뢰도 정보가 전달됩니다.
