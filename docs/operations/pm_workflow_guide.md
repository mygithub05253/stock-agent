# PM/기획자를 위한 AI 기반 프로젝트 문서 및 대시보드 관리 방법론 (30단계)

| 항목 | 값 | 
| ----- | ----- | 
| 작성자 | AI Assistant (PM 실무 벤치마킹 기반) | 
| 대상 독자 | 코드 작업 없이 프로젝트/문서를 총괄하는 PM 및 기획자 | 
| 목적 | 대시보드 기획, HTML/MD 활용 전략, MCP/AI 연동 실무 가이드 제공 | 
| 상위 문서 | `docs/README.md` | 

## 📌 Section 1: 기획 문서 및 대시보드 관리 실무 (Step 1 ~ Step 10)

실무에서 프로젝트 대시보드는 단순한 정보 나열이 아닌, '의사결정을 지원하는 도구'로 활용됩니다. 글로벌 IT 기업(Lark, Monday.com 등)의 벤치마킹 결과를 바탕으로 대시보드 및 명세서 관리 체계를 구축합니다.

* **Step 1. 대시보드의 목적(Goal) 정의:** 대시보드가 어떤 의사결정(예: 일정 지연 파악, 리스크 감지)을 지원할지 명확히 합니다.
* **Step 2. 대상 독자(Stakeholder) 분류:** 개발팀, 데이터팀, 이해관계자 등 보는 사람에 따라 필요한 정보 깊이를 나눕니다.
* **Step 3. 핵심 성과 지표(KPI) 및 임계값 설정:** 프로젝트 진행률, 남은 티켓 수, LLM 비용 소진율(월 5만원 상한 등) 등 추적할 핵심 지표를 선정합니다.
* **Step 4. 3단계 정보 계층화 (Progressive Disclosure):**
  * *Tier 1 (요약):* 현재 상태 및 긴급 이슈 (시각적 대시보드)
  * *Tier 2 (세부 지표):* 주요 마일스톤 및 티켓별 진행률
  * *Tier 3 (원천 데이터):* PRD, 기능 명세서 등 개별 Markdown 문서 링크
* **Step 5. 단일 진입점(Single Source of Truth) 구축:** 현재 `docs/README.md`나 `docs/roadmap/` 하위의 HTML 대시보드처럼 모든 팀원이 가장 먼저 열어보는 인덱스 페이지를 유지합니다.
* **Step 6. 데이터 소스 중앙화 및 테이블화:** 산재된 기획 내용(요구사항, API, DB 스키마 등)을 표(Table) 형태로 정리하여 AI와 팀원이 구조적으로 읽기 쉽게 만듭니다.
* **Step 7. 시각화 도구 활용:** Mermaid.js 등을 Markdown 내에 적극 활용하여, 흐름도나 ERD를 코드로 관리하면서 시각적 효과를 얻습니다. (현재 `system_flow.md`에 잘 적용되어 있습니다.)
* **Step 8. 명세서(Spec)의 모듈화:** PRD 하나에 모든 것을 담지 않고, 기능 명세(`functional-spec`), 구조(`architecture`) 등으로 파일을 쪼개어 관리합니다.
* **Step 9. 리뷰 및 거버넌스 사이클 설정:** 주간 회의 전 PM이 대시보드와 명세서의 최신화 상태를 점검하는 루틴을 만듭니다.
* **Step 10. 형상 관리(Git) 동기화:** 기획 문서도 개발 코드처럼 PR(Pull Request)을 통해 리뷰하고 병합하여 히스토리를 보존합니다.

## 📌 Section 2: Markdown vs HTML 실무 활용 전략 (Step 11 ~ Step 20)

최근 테크니컬 라이팅 실무에서는 "빠른 작성과 협업은 Markdown으로, 정밀한 표현과 대시보드는 HTML/CSS로" 구분하는 것이 트렌드입니다.

* **Step 11. Markdown의 핵심 용도 정의:** PRD, 기능 명세서, 회의록 등 텍스트 중심의 '콘텐츠(Content)' 작성에 집중합니다. 작성 속도가 빠르고 Git에서 Diff(변경점)를 보기가 가장 좋습니다.
* **Step 12. HTML의 핵심 용도 정의:** 현재 레포의 `roadmap_dashboard.html`이나 `system_architecture_dashboard.html`처럼 인터랙티브 요소(탭, 접기/펴기, 검색)가 필요하거나 레이아웃(디자인)이 중요한 '프레젠테이션(Presentation)'에 사용합니다.
* **Step 13. 작성 공정의 분리 (Drafting vs Rendering):** 기획자는 내용을 Markdown으로 초안을 작성합니다. 디자인이 필요한 최종 대시보드 형태는 AI를 이용해 HTML로 변환/렌더링합니다.
* **Step 14. 컴포넌트 단위 설계 사고:** 문서도 UI 컴포넌트처럼 생각합니다. "이 정책 블록은 다른 문서에서도 쓰이니까 분리하자"라는 마인드셋을 가집니다.
* **Step 15. Markdown 한계 인식:** Markdown은 복잡한 UI, 커스텀 버튼, 보안/접근성 태그를 지원하지 못함을 인지하고 무리하게 MD로 디자인을 구현하려 하지 않습니다.
* **Step 16. 하이브리드 활용:** Markdown 파일 내에 `<details>`, `<summary>` 같은 간단한 HTML 태그를 섞어 사용하여 문서의 가독성을 높입니다.
* **Step 17. AI 친화적 포맷팅:** AI(Codex 등)가 읽고 파싱하기 쉽도록 들여쓰기, 불릿 포인트, 헤더(`#`)의 계층 구조를 엄격하게 지킵니다.
* **Step 18. 폴더별 README 작성 (현재 규칙 엄수):** 각 문서 폴더(`docs/prd/`, `docs/architecture/` 등)마다 `README.md`를 두어 해당 폴더의 작성 규칙과 템플릿을 명시합니다.
* **Step 19. 일관된 Naming Rule:** 문서 파일명에 버전(예: `PRD_v0.6.md`)과 날짜를 명시하여, HTML 기반 대시보드에서 링크를 걸기 쉽게 유지합니다.
* **Step 20. 정적 사이트 제너레이터(SSG) 고려:** 향후 프로젝트가 커지면 Markdown을 모아 자동으로 예쁜 HTML 웹사이트(Docs)로 구워주는 도구(예: Docusaurus, MkDocs) 도입을 고려합니다.

## 📌 Section 3: AI 어시스턴트(Codex) 및 MCP 연동 실무 (Step 21 ~ Step 30)

MCP(Model Context Protocol)는 AI가 외부의 데이터(문서, DB, Jira 등)와 안전하게 연결되어 작업할 수 있게 해주는 최신 표준입니다. 현재 Codex 환경에서 기획자가 AI를 200% 활용하는 커스터마이징 방법입니다.

* **Step 21. AI에게 역할(Persona) 부여:** 문서 작성을 요청할 때 "너는 10년 차 IT 프로젝트 매니저이자 테크니컬 라이터야"라고 명시합니다.
* **Step 22. 기존 규칙 주입 (Context Feeding):** 문서 작업을 요청할 때 **반드시** 루트 `README.md`와 `docs/README.md`의 내용을 먼저 읽고 그 규칙(브랜치 규칙, 용어, 비용 5만원 상한 등)을 따르도록 지시합니다.
* **Step 23. 시스템 프롬프트(System Prompt) 템플릿화:** 자주 하는 요청(예: "기능 명세서 초안 작성")은 사전에 템플릿화하여 AI에게 전달합니다.
* **Step 24. MCP 개념의 실무 적용 (가상화):** 직접적인 MCP 도구 연결이 제한된 환경이라면, 기획자가 "Jira 티켓 내용", "DB 스키마 내용"을 복사해서 AI의 컨텍스트 창에 직접 붙여넣어(프롬프팅) 가상의 MCP 환경을 만듭니다.
* **Step 25. 단계별 문서 생성 요청 (Chain of Thought):** AI에게 한 번에 완벽한 PRD를 요구하지 않습니다. "1. 목차를 짜줘 -> 2. 1번 항목을 써줘 -> 3. 피드백 반영해줘" 식으로 나누어 작업합니다.
* **Step 26. 다이어그램 코드 생성 자동화:** 로직이나 아키텍처가 바뀌면 AI에게 "업데이트된 요구사항을 바탕으로 Mermaid.js 코드를 다시 짜줘"라고 요청하여 다이어그램을 최신화합니다.
* **Step 27. 용어집(Glossary) 교차 검증:** 새 문서를 작성할 때마다 AI에게 "`docs/glossary.md`에 없는 새로운 용어가 쓰였는지 확인하고 표로 정리해줘"라고 요청하여 일관성을 유지합니다.
* **Step 28. 에지 케이스(Edge Case) 도출 활용:** 기획안을 작성한 후 AI에게 "이 기능에서 발생할 수 있는 에지 케이스나 예외 상황을 5가지 찾아줘"라고 요청하여 기획의 빈틈을 메웁니다.
* **Step 29. HTML 대시보드 갱신 자동화:** 주간 로드맵이나 현황이 바뀌면, 변경된 데이터(Markdown)를 AI에게 주고 "이 데이터를 바탕으로 `roadmap_dashboard.html` 코드를 갱신해줘"라고 요청합니다.
* **Step 30. AI 기반 산출물 검수:** 완성된 문서에 대해 "우리 팀의 커밋 규칙과 문서화 가이드라인(`docs/README.md`)에 위배되는 사항이 있는지 검수해줘"라고 최종 리뷰를 맡깁니다.

## 🎯 결론 및 현재 프로젝트 적용 방안

PM님, 현재 `stock-agent` 레포지토리는 이미 `docs/architecture/system_architecture_dashboard.html`과 같이 훌륭한 하이브리드 문서화(MD+HTML) 체계를 갖추고 있습니다.

앞으로 **Codex(AI)와 협업하실 때 가장 중요한 팁**은 새로운 문서를 생성하거나 편집하실 때 대화창에 **"루트 경로의 `README.md`와 `docs/README.md`에 명시된 정책(폴더 분담, LLM 비용 상한 등)을 반드시 지켜서 작성해 줘"** 라는 지시문을 습관적으로 포함시키는 것입니다. 이렇게 하면 AI가 프로젝트의 규칙을 절대 어기지 않고 실무 전문가 수준의 기획 문서와 대시보드 코드를 생성해 낼 것입니다.
