PM Opinion

내가 동의하는 것과 우려하는 것

동의하는 것

  • MVP는 central supervisor 구조가 맞다.
  • Quant/Qual/Competitor를 specialist로 나눈 것은 좋다.
  • 계산은 tools/DB, 정성 근거는 RAG, 종합은 Strategist, 검수는 Guardrail로 나누는 방향이 맞다.
  • Prompt와 schema를 분리하는 방향은 PM 리뷰에도 도움이 된다.

우려하는 것

  • AgentState가 아직 PB report와 UI까지 고려한 계약인지 불명확하다.
  • Guardrail이 구체적인 차단/완화 정책 없이 구호로 남을 수 있다.
  • Evidence contract가 없으면 Tier 3 보고서의 설득력이 약해진다.
  • 일부 agent 실패 시 전체를 막을지 warning으로 진행할지 정책이 필요하다.

Plain Language

어려운 용어 쉽게 풀기

AgentState

agent들이 함께 쓰는 공용 작업일지입니다. 이 일지가 부실하면 UI와 PB report도 부실해집니다.

Evidence bundle

보고서에 붙는 영수증입니다. 어떤 근거로 말했는지 확인할 수 있어야 합니다.

Guardrail

사용자에게 결과가 나가기 전 마지막 검수대입니다. 위험한 표현과 출처 없는 단정을 막습니다.

Faithfulness

RAG가 찾은 근거 안에 있는 말만 했는지 보는 지표입니다. hallucination 방지의 핵심입니다.

Decision Tickets

구현 전에 닫아야 할 이슈

NEED-DECISION-001: [Architecture] AgentState와 report-ready result schema 확정

request_id, as_of_date, data_version, mode, evidence_bundle, cost_trace, errors, warnings를 포함할지 결정한다. AgentState는 공용 작업일지다.

NEED-DECISION-002: [Architecture] Evidence bundle과 source attachment contract 확정

source_type, title, url/document_id, published_at, chunk_id, retrieval_score를 포함한 source contract를 확정한다. Evidence bundle은 보고서의 영수증이다.

NEED-DECISION-003: [Guardrail] Input/Tool/Output Guardrail enforcement plan 확정

무엇을 block하고, 무엇을 warning으로 통과시키고, disclaimer를 언제 붙일지 결정한다. Guardrail은 마지막 검수대다.

NEED-DECISION-004: [LLM] Cost trace와 월 5만원 상한 정책 구현 기준 확정

agent별 token/cost, cache hit, 3만원/4만원/5만원 임계 동작을 정한다. 비용 추적은 가계부다.

NEED-DECISION-005: [Architecture] Partial Success with Warnings 정책 채택 여부 결정

Quant 실패처럼 일부 실패는 warning으로 진행하고, Strategist/Guardrail 실패는 block할지 결정한다.

NEED-DECISION-006: [Evaluation] MVP evaluation gate 확정

RAG Faithfulness를 Qual hallucination 방지의 최우선 gate로 둔다. 검색 근거에 없는 말은 만들면 안 된다.

Recommended Failure Policy

Partial Success with Warnings

상황 내 제안
Quant 실패, Qual/Competitor 성공 분석 진행. 정량 근거 부족 warning과 confidence 하향.
Qual RAG 실패 정성 요약 생성 금지. source 부족 warning 표시.
Competitor 실패 Peer 비교만 제외하고 진행.
Strategist 실패 최종 signal 생성 불가. 결과 block.
Guardrail 실패 사용자-facing 출력 block.

Meeting Script

회의에서 내가 말할 문장

저는 PR #20의 큰 방향에는 동의합니다.
MVP에서는 Curator -> Quant/Qual/Competitor 병렬 -> Strategist -> Guardrail 구조를 기준선으로 잡는 게 맞다고 봅니다.

다만 이것은 구현 승인이라기보다는 architecture baseline 승인입니다.
구현 전에 AgentState, evidence contract, Guardrail enforcement, cost trace, partial failure policy, evaluation gate는 별도 이슈로 닫고 가야 합니다.

특히 MVP에서 A2A/full decentralized agents는 제외했으면 합니다.
지금 우리 구조는 Streamlit + Postgres + 단일 repo이기 때문에 central supervisor가 더 안전합니다.

또 하나 중요한 건 RAG Faithfulness입니다.
Qual agent가 검색된 근거에 없는 내용을 만들어내면 금융 분석 제품에서는 가장 위험한 문제가 됩니다.
그래서 Ragas Faithfulness를 MVP 최우선 evaluation gate로 두는 것을 제안합니다.
