한국 주식 투자자를 위한 PB 보조형 멀티 에이전트 분석 시스템

이 문서는 현재 프로젝트의 PRD, ADR, system flow, Phase 1 pipeline을 바탕으로 멀티 에이전트의 큰 구조, 각 agent의 책임, LangGraph 흐름, DB 연결, prompt 운영, guardrail과 평가 지표를 한 번에 논의하기 위해 정리한 자료입니다.

Manager / Supervisor Pattern Postgres + pgvector Evidence-first Output Financial Guardrails

한 줄 결론

현재 프로젝트는 완전 분산형 A2A보다, Curator가 요청을 구조화하고, Quant/Qual/Competitor가 병렬 분석한 뒤, Strategist가 종합하고 Guardrail이 검증하는 Supervisor graph가 가장 안전합니다.

Shared Contract

AgentState가 시스템의 중심

멀티 에이전트에서 가장 중요한 것은 agent끼리 자유 대화하게 하는 것이 아니라, 모든 agent가 같은 state contract를 읽고 쓰도록 만드는 것입니다.

필드역할필요 이유
request_id분석 요청 고유 ID로그, 비용, 평가, 재현성 연결
as_of_date분석 기준일백테스트와 미래 데이터 차단
data_version데이터 스냅샷 버전동일 결과 재현
evidence_bundle모든 출처 묶음출처 부착률, hallucination 검사
cost_traceagent별 비용/토큰월 5만원 비용 상한 관리
errors / warnings부분 실패와 경고Tier 1 부분 응답과 운영 디버깅

Agent Responsibilities

세부 에이전트 작동 방식

Curator Agent

입구 / 라우터

Input

user_query, user_profile, portfolio, company DB

Work

종목명/코드 추출, 의도 분류, 후보 추천, KRX stock_code 확정

Output

intent, stock_code, corp_code, corp_name, sector, candidates

Prompt 원칙: 모르면 추측하지 말고 후보를 반환한다. KRX 6자리 종목코드와 DART corp_code를 함께 반환한다.

Quant Agent

정량 / 계산

Input

stock_code, corp_code, stock_price, financial_statement, raw_macro

Work

PER/PBR/ROE, 성장률, 5년 valuation, 안전마진, volatility 계산

Output

quant_score, valuation_signal, metrics, valuation_range, reasons, risks

핵심 정책: 숫자는 LLM이 만들지 않는다. 계산은 Python/DB가 하고, LLM은 계산 결과 해석만 한다.

Qual Agent

뉴스 / 공시 RAG

Input

stock_code, corp_code, query intent, rag_documents, rag_chunks

Work

pgvector 검색, 호재/악재 분류, 이벤트 유형 분류, 출처 기반 요약

Output

sentiment, qual_score, event_types, evidence, source_urls, risks

핵심 정책: 검색된 chunk에 없는 사실은 말하지 않는다. RAG 결과가 약하면 confidence를 낮춘다.

Competitor Agent

Peer 비교

Input

target stock_code, sector, company, stock_price, financial_statement

Work

동종업계 peer 선정, PER/PBR/ROE/성장률 비교, 상대 위치 판단

Output

peer_list, peer_metrics, relative_position, competitor_score, evidence

핵심 정책: peer 선정 기준을 설명 가능하게 남긴다. 글로벌 peer는 MVP 범위 밖이면 제외한다.

Strategist Agent

Supervisor / 종합

Input

Curator, Quant, Qual, Competitor 결과 + user_profile + portfolio

Work

정량/정성/Peer 종합, 포트 적합도 계산, 5가지 관점 self-critique

Output

signal, confidence, suitability, headline, key_reasons, risks, next_actions

핵심 정책: signal과 suitability를 분리한다. 종목은 좋아도 이미 비중이 높으면 suitability는 낮을 수 있다.

Guardrail & Evaluator

안전 / 평가

Input

strategist_result, evidence_bundle, user_query, policy rules

Work

투자 권유성 표현 완화, 수익 보장 차단, 출처 없는 주장 검사, 평가 점수 기록

Output

passed, warnings, revised_headline, disclaimer, eval_scores

핵심 정책: 마지막 문장 다듬기가 아니라 금융 시스템의 runtime control layer로 설계한다.

Data Architecture

어떤 DB에서 무엇을 가져오는가

MVP 기본 방향은 Postgres + pgvector 단일 DB입니다. 정형 데이터는 일반 table, 뉴스/공시 원문과 embedding은 rag_documents, rag_chunks에서 관리합니다.

Prompt Operations

프롬프트는 코드 밖에서 관리

PM과 기획자가 함께 관리할 수 있도록 agent별 prompt를 markdown으로 분리합니다. 모든 prompt는 구조화 출력, 근거 제한, 모르면 모른다고 답하기, 금융 표현 제한을 포함해야 합니다.

Role

agent가 맡은 책임을 한 문장으로 고정합니다.

Data Boundary

사용 가능한 데이터와 금지된 추정을 명시합니다.

Schema

Pydantic/JSON schema에 맞춰 반환하게 합니다.

Safety

투자 권유, 수익 보장, 출처 없는 단정을 금지합니다.

Tools Boundary

agents/와 tools/ 책임 분리

agents/에는 판단 흐름을, tools/에는 DB 조회, API 호출, 계산식을 둡니다. 이렇게 해야 agent prompt를 바꿔도 데이터 로직이 흔들리지 않습니다.

팀 논의용 최종 문장

우리는 6-agent 구조를 유지한다. Curator는 질문을 분석 요청으로 바꾸고, Quant/Qual/Competitor는 병렬로 전문 분석을 수행하고, Strategist는 사용자 포트폴리오 맥락으로 종합하며, Guardrail은 금융 표현과 근거 품질을 최종 검증한다. 모든 agent는 schema로 대화하고, 모든 수치 계산은 Python/DB가 하며, 모든 정성 판단은 RAG evidence를 가져야 한다.
