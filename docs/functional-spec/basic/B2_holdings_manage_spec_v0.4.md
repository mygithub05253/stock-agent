## 0. 문서 메타 정보

본 문서는 PRD 기준 기본 기능 `B2. 보유 종목 등록/조회`의 상세 기능 명세서다. 기존 `overview/functional_spec_all_features_v0.1.md`의 7대 표준 양식인 **트리거 / 전제조건 / 입력 / 처리 흐름 / 출력 / 예외 처리 / 담당**을 유지하고, `basic/B5_portfolio_bulk_advice_spec_v0.2.md`에서 확립한 비용 상한 및 데이터 결손 예외 처리 기준을 동일하게 적용한다.

`B2`는 사용자의 포트폴리오 원장 데이터를 만드는 기능이다. 이후 `B5. 포트폴리오 일괄 안내`, `A4. BUY/HOLD/SELL 권유`, `A5. 종목 추천`이 모두 `holdings` 데이터를 기반으로 동작하므로, 단순 CRUD처럼 보이더라도 입력 검증, 중복 처리, CSV 업로드 오류 표시, 데이터 품질 관리가 중요하다.

## 2. 기능 개요

보유 종목 등록/조회 기능은 로그인한 사용자가 자신의 보유 종목, 평균 매수가, 수량, 매수일, 메모를 직접 입력하거나 CSV로 일괄 업로드하고, 저장된 포트폴리오 목록을 조회하는 기능이다.

MVP에서는 한국 상장 종목을 대상으로 하며, `company` 테이블에 존재하는 종목만 등록 가능하다. 사용자는 종목별 평가금액, 평가손익, 수익률, 포트폴리오 내 비중을 확인할 수 있다. 실시간 주문 연동이나 증권사 계좌 자동 동기화는 제외하고, 사용자가 입력한 값을 기준 원장으로 사용한다.

## 4. B2. 보유 종목 등록/조회

### 4.1 트리거

- 로그인한 사용자가 `보유 종목 관리` 메뉴를 클릭한다.
- 사용자가 `종목 추가` 버튼을 클릭해 단일 보유 종목 등록 폼을 연다.
- 사용자가 CSV 파일을 업로드하고 `일괄 등록` 버튼을 클릭한다.
- 사용자가 보유 종목 표에서 특정 종목의 `수정` 또는 `삭제` 버튼을 클릭한다.
- B5 포트폴리오 일괄 안내 또는 A4 BUY/HOLD/SELL 권유 화면이 현재 보유 종목 목록을 조회한다.

### 4.2 전제조건

| 조건 | 상세 |
|------|------|
| 로그인 | `B1` 로그인 완료 상태이며 세션에서 `user_id`를 확인할 수 있어야 한다. |
| 사용자 테이블 | `users` 테이블에 현재 `user_id`가 존재해야 한다. |
| 보유 종목 테이블 | Postgres에 `holdings` 테이블이 존재해야 한다. |
| 종목 마스터 | `company` 테이블에 등록 가능한 종목의 `stock_code`, `corp_name`, `sector`가 적재되어 있어야 한다. |
| 시세 데이터 | 조회 화면에서 평가손익을 계산하려면 `stock_price`에 최근 가격이 있어야 한다. 단, 등록 자체는 시세 데이터가 없어도 가능하다. |
| CSV 파서 | CSV 업로드 시 지정 헤더를 검증할 수 있어야 한다. |
| 비용 정책 | B2는 LLM 호출 없이 동작해야 하며 `llm_cost_log`에 비용을 발생시키지 않는다. |

### 4.3 입력

단일 등록/수정 입력:

| 필드 | 타입 | 필수 | 검증 기준 | 설명 |
|------|------|------|-----------|------|
| `user_id` | uuid/string | 필수 | 세션에서 획득 | 현재 로그인 사용자 |
| `stock_code` | string(6) | 필수 | 숫자 6자리, `company.stock_code` 존재 | 보유 종목 코드 |
| `avg_price` | int/decimal | 필수 | 0보다 큼, 원화 기준 | 평균 매수가 |
| `qty` | int | 필수 | 1 이상 정수 | 보유 수량 |
| `bought_at` | date | 선택 | 미래 날짜 불가 | 대표 매수일 또는 최초 매수일 |
| `memo` | text | 선택 | 500자 이하, PII/스크립트 제거 | 사용자 메모 |
| `source` | enum | 선택 | `manual`, `csv`, `demo` | 데이터 입력 경로 |

CSV 업로드 입력:

| 컬럼 | 타입 | 필수 | 검증 기준 | 설명 |
|------|------|------|-----------|------|
| `stock_code` | string(6) | 필수 | 숫자 6자리, `company.stock_code` 존재 | 종목 코드 |
| `avg_price` | int/decimal | 필수 | 0보다 큼 | 평균 매수가 |
| `qty` | int | 필수 | 1 이상 정수 | 보유 수량 |
| `bought_at` | date | 선택 | YYYY-MM-DD, 미래 날짜 불가 | 매수일 |
| `memo` | text | 선택 | 500자 이하 | 사용자 메모 |

조회 입력:

| 필드 | 타입 | 필수 | 출처 | 설명 |
|------|------|------|------|------|
| `user_id` | uuid/string | 필수 | 세션 | 현재 사용자 보유 종목 조회 |
| `sort_by` | enum | 선택 | UI | `weight`, `return_rate`, `corp_name`, `updated_at` 등 |
| `filter_sector` | string | 선택 | UI | 특정 섹터만 조회 |
| `include_price` | boolean | 선택 | UI/시스템 | 최근 시세 및 평가손익 포함 여부 |

시스템 생성 값:

| 필드 | 타입 | 생성 시점 | 설명 |
|------|------|-----------|------|
| `holding_id` | uuid/bigint | 신규 등록 시 | 보유 종목 행 식별자 |
| `created_at` | datetime | 신규 등록 시 | 생성 시각 |
| `updated_at` | datetime | 수정 시 | 최근 수정 시각 |
| `weight` | decimal | 조회/집계 시 | 포트폴리오 내 평가금액 비중 |
| `market_value` | decimal | 조회/집계 시 | 최근 가격 기준 평가금액 |
| `unrealized_pnl` | decimal | 조회/집계 시 | 평가손익 |
| `return_rate` | decimal | 조회/집계 시 | 평가수익률 |

### 4.4 처리 흐름

1. **보유 종목 관리 화면 진입**
   - 사용자가 `보유 종목 관리` 메뉴를 클릭하면 세션의 `user_id`를 확인한다.
   - 로그인 상태가 아니면 B1 로그인 화면으로 이동한다.
   - 로그인 상태이면 `holdings`에서 사용자 보유 종목을 조회한다.

2. **보유 종목 목록 조회**
   - `holdings`를 `user_id` 기준으로 조회한다.
   - `company`와 조인해 종목명, 섹터, 상장 상태를 표시한다.
   - `include_price=true`이면 `stock_price`의 최근 가격을 조인하여 평가금액, 평가손익, 수익률을 계산한다.
   - 시세 데이터가 없는 종목은 `시세 없음` 상태로 표시하되, 보유 정보는 그대로 보여준다.

3. **포트폴리오 비중 계산**
   - 각 종목의 `market_value = current_price * qty`를 계산한다.
   - 최근 가격이 없는 종목은 비중 계산에서 제외하거나 `price_missing` 상태로 분리한다.
   - 전체 평가금액이 0보다 크면 `weight = market_value / total_market_value`를 계산한다.
   - 계산은 LLM이 아니라 Python/SQL로 수행한다.

4. **단일 종목 등록 폼 입력**
   - 사용자는 종목 검색/자동완성 또는 직접 입력으로 `stock_code`를 선택한다.
   - 평균 매수가, 수량, 매수일, 메모를 입력한다.
   - 종목 검색 자동완성은 B3 기능과 연계되며, B2 단독 MVP에서는 `company` 테이블 조회로 최소 구현 가능하다.

5. **단일 입력 검증**
   - `stock_code`가 숫자 6자리인지 확인한다.
   - `company` 테이블에 해당 `stock_code`가 존재하는지 확인한다.
   - `avg_price > 0`, `qty >= 1`인지 확인한다.
   - `bought_at`이 미래 날짜인지 확인한다.
   - `memo`는 500자 이하로 제한하고 스크립트/제어문자를 제거한다.

6. **MVP 지원 범위 검증**
   - MVP 관심 섹터 또는 지원 대상 섹터가 정해져 있으면 `company.sector`를 기준으로 경고를 표시한다.
   - 지원 범위 밖 종목도 저장할지 여부는 운영 정책에 따른다. MVP 기본값은 저장 가능하되 `지원 데이터 제한` 경고를 표시한다.

7. **중복 종목 처리**
   - 동일 `user_id`, `stock_code` 조합이 이미 존재하는지 확인한다.
   - 기존 행이 있으면 기본 동작은 갱신 모드로 전환한다.
   - 갱신 시 평균 매수가와 수량은 사용자가 입력한 최종 값으로 덮어쓴다.
   - 향후 확장에서는 매수 lot 단위 이력 테이블을 둘 수 있으나 MVP에서는 종목별 1행 원장 방식을 사용한다.

8. **DB 저장**
   - 신규 종목이면 `holdings`에 INSERT한다.
   - 기존 종목이면 `holdings`를 UPDATE한다.
   - 저장 후 `updated_at`을 갱신한다.
   - 저장 성공 이벤트는 PII 없이 `holding_upsert_success`로 기록한다.

9. **CSV 업로드 접수**
   - 사용자가 CSV 파일을 업로드하면 헤더를 먼저 검증한다.
   - 허용 헤더는 `stock_code`, `avg_price`, `qty`, `bought_at`, `memo`다.
   - 파일 크기와 행 수 제한을 둔다. MVP 권장값은 200행 이하, 1MB 이하이다.

10. **CSV 행별 검증**
    - 각 행에 대해 단일 입력 검증과 동일한 규칙을 적용한다.
    - 오류가 있는 행은 저장 대상에서 제외하고, 행 번호와 오류 사유를 결과 표에 표시한다.
    - 유효한 행만 INSERT/UPDATE 후보로 분류한다.

11. **CSV 일괄 저장**
    - 유효 행은 트랜잭션으로 일괄 저장한다.
    - 일부 행 오류가 있어도 유효 행 저장을 허용하는 `partial_success` 방식을 기본값으로 한다.
    - DB 장애가 발생하면 전체 트랜잭션을 롤백하고 저장 실패 메시지를 표시한다.

12. **삭제 처리**
    - 사용자가 보유 종목 표에서 `삭제`를 클릭하면 확인 모달을 표시한다.
    - 확인 시 `holdings`에서 해당 `holding_id` 또는 `user_id + stock_code` 행을 삭제한다.
    - 삭제 후 목록과 비중을 다시 계산한다.

13. **UI 갱신**
    - 저장/수정/삭제/CSV 업로드 완료 후 보유 종목 표를 즉시 갱신한다.
    - 종목별 평가손익, 수익률, 비중, 섹터 분포를 다시 계산해 표시한다.
    - B5 일괄 안내로 이동할 수 있는 버튼을 제공한다.

14. **비용/LLM 처리**
    - B2는 보유 데이터 CRUD 기능이므로 LLM을 호출하지 않는다.
    - 종목명 보정, 메모 정리, CSV 오류 설명도 규칙 기반 문구로 처리한다.
    - `llm_cost_log`에는 B2 관련 비용 로그를 남기지 않는다.

| 출력 대상 | 필드/요소 | 타입 | 설명 |

`holdings` 테이블 권장 필드:

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `holding_id` | uuid/bigint | 필수 | PK |

조회 계산 값:

| 필드 | 계산식 | 설명 |
|------|--------|------|
| `cost_basis` | `avg_price * qty` | 투자원금 |
| `market_value` | `current_price * qty` | 평가금액 |
| `unrealized_pnl` | `(current_price - avg_price) * qty` | 평가손익 |
| `return_rate` | `(current_price - avg_price) / avg_price` | 평가수익률 |
| `weight` | `market_value / total_market_value` | 포트폴리오 비중 |

### 4.6 예외 처리

| 예외 상황 | 감지 조건 | 처리 방안 | 사용자 메시지 |
|-----------|-----------|-----------|---------------|
| 미로그인 | 세션에 `user_id` 없음 | B1 로그인 화면으로 이동 | `로그인이 필요합니다.` |
| 보유 종목 0개 | `holdings` 조회 결과 0건 | 빈 상태 화면과 등록 CTA 표시 | `보유 종목을 추가해 주세요.` |
| 종목 코드 형식 오류 | `stock_code`가 6자리 숫자가 아님 | 저장 차단, 폼 유지 | `종목코드는 6자리 숫자로 입력해 주세요.` |
| 지원하지 않는 종목 | `company.stock_code` 조회 실패 | 저장 차단 또는 지원 불가 안내 | `지원하지 않는 종목입니다. 종목 검색에서 다시 선택해 주세요.` |
| 평균 매수가 오류 | `avg_price <= 0` | 저장 차단 | `평균 매수가는 0보다 커야 합니다.` |
| 수량 오류 | `qty < 1` 또는 정수 아님 | 저장 차단 | `수량은 1주 이상 정수로 입력해 주세요.` |
| 미래 매수일 | `bought_at > today` | 저장 차단 또는 날짜 제거 | `매수일은 오늘 이후 날짜로 입력할 수 없습니다.` |
| 메모 길이 초과 | `len(memo) > 500` | 저장 차단 또는 500자 이하로 요구 | `메모는 500자 이하로 입력해 주세요.` |
| 메모 내 스크립트 | `<script>` 등 위험 문자열 감지 | 위험 문자열 제거 후 저장 또는 저장 차단 | `메모에 허용되지 않는 문자가 포함되어 있습니다.` |
| 중복 종목 | 동일 `user_id + stock_code` 존재 | 갱신 확인 후 UPDATE | `이미 등록된 종목입니다. 기존 정보를 업데이트할까요?` |
| CSV 헤더 오류 | 필수 헤더 누락/오타 | 업로드 중단, 예시 헤더 표시 | `CSV 헤더를 확인해 주세요: stock_code, avg_price, qty, bought_at, memo` |
| CSV 행 오류 | 특정 행 검증 실패 | 해당 행 제외, 행 번호와 사유 표시 | `일부 행은 형식 오류로 제외되었습니다.` |
| CSV 파일 과대 | 200행 초과 또는 1MB 초과 | 업로드 차단 | `CSV 파일은 200행, 1MB 이하로 업로드해 주세요.` |
| CSV 전체 실패 | 유효 행 0건 | 저장 없음, 오류 요약 표시 | `저장 가능한 행이 없습니다. CSV 내용을 확인해 주세요.` |
| 일부 저장 성공 | 유효 행과 오류 행 혼재 | 유효 행만 저장, 실패 행 표기 | `일부 종목만 저장되었습니다. 실패 사유를 확인해 주세요.` |
| DB 저장 실패 | INSERT/UPDATE 예외 | 트랜잭션 롤백, 에러 로그 저장 | `일시적으로 저장할 수 없습니다. 잠시 후 다시 시도해 주세요.` |
| 삭제 대상 없음 | 삭제 요청 행이 이미 없음 | 목록 새로고침 | `이미 삭제되었거나 존재하지 않는 보유 종목입니다.` |
| 시세 데이터 없음 | `stock_price` 최근 가격 없음 | 등록은 유지, 평가손익은 `시세 없음` 표시 | `최신 시세가 없어 평가손익을 계산하지 못했습니다.` |
| LLM 호출 발생 위험 | B2 처리 중 LLM 라우터 호출 시도 | 호출 차단, 개발 로그에 정책 위반 기록 | 사용자 노출 없음 |

- B2는 사용자가 입력한 포트폴리오 원장이므로 저장 전 검증을 엄격히 한다.

- 증권사 계좌 자동 연동은 MVP 범위에서 제외한다.
