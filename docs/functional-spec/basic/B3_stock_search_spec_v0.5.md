## 0. 문서 메타 정보

본 문서는 PRD 기준 기본 기능 `B3. 종목 검색`의 상세 기능 명세서다. 기존 `overview/functional_spec_all_features_v0.1.md`의 7대 표준 양식인 **트리거 / 전제조건 / 입력 / 처리 흐름 / 출력 / 예외 처리 / 담당**을 유지하고, 기존 B1/B2/B5 명세에서 정의한 비용 상한, 입력 검증, 데이터 결손 예외 처리 기준을 동일하게 적용한다.

`B3`는 사용자가 종목명 또는 종목코드를 정확히 외우지 않아도 분석 가능한 종목을 빠르게 찾게 하는 진입 기능이다. B2 보유 종목 등록, B4 종목 기본 정보 조회, A1~A4 분석 실행의 앞단에 놓이므로 검색 정확도와 응답 속도가 중요하다.

## 2. 기능 개요

종목 검색 기능은 사용자가 검색창에 종목명, 종목코드, 일부 키워드를 입력하면 `company` 테이블을 기준으로 분석 가능한 종목 후보를 자동완성 형태로 보여주는 기능이다.

MVP에서는 `company` 테이블에 적재된 한국 상장 종목을 대상으로 하며, 검색 결과에는 종목명, 종목코드, 섹터, 최근 가격, 등락률, 지원 범위 상태를 표시한다. 검색 결과를 클릭하면 B4 종목 기본 정보 조회 화면으로 이동하거나 B2 보유 종목 등록 폼에 종목이 자동 입력된다.

## 4. B3. 종목 검색

### 4.1 트리거

- 사용자가 홈 화면, 종목 상세 화면, 보유 종목 등록 폼, 또는 포트폴리오 화면의 검색창에 종목명 일부를 입력한다.
- 사용자가 검색창에 종목코드 일부 또는 전체 6자리를 입력한다.
- 사용자가 B2 보유 종목 등록 중 `종목 선택` 자동완성을 연다.
- 사용자가 B4 종목 기본 정보 조회를 위해 종목 검색 결과를 클릭한다.
- 사용자가 검색어 입력 후 Enter 또는 검색 버튼을 누른다.

### 4.2 전제조건

| 조건 | 상세 |
|------|------|
| 종목 마스터 | `company` 테이블에 `stock_code`, `corp_name`, `sector`가 적재되어 있어야 한다. |
| 검색 인덱스 | `company.stock_code`, `company.corp_name`에 검색 성능을 위한 인덱스 또는 LIKE 최적화가 필요하다. |
| 시세 데이터 | 결과에 현재가/등락률을 표시하려면 `stock_price`에 최근 가격이 있어야 한다. 단, 검색 자체는 시세 없이 가능하다. |
| UI 컴포넌트 | 검색창, 자동완성 드롭다운, 결과 선택 이벤트를 처리할 수 있어야 한다. |
| 로그인 | 기본 검색은 비로그인도 가능하나, B2 등록 연결은 로그인 상태가 필요하다. |
| 비용 정책 | B3는 LLM 호출 없이 동작해야 하며 `llm_cost_log`에 비용을 발생시키지 않는다. |

### 4.3 입력

검색 입력:

| 필드 | 타입 | 필수 | 검증 기준 | 설명 |
|------|------|------|-----------|------|
| `query` | string | 필수 | trim 후 2글자 이상 또는 종목코드 2자리 이상 | 사용자가 입력한 검색어 |
| `search_type` | enum | 선택 | `auto`, `name`, `code` | 기본값 `auto` |
| `limit` | int | 선택 | 1~20, 기본값 10 | 반환 결과 수 상한 |
| `include_price` | boolean | 선택 | 기본값 `true` | 최근 가격/등락률 포함 여부 |
| `include_sector_filter` | boolean | 선택 | 기본값 `false` | 사용자 관심 섹터 우선 정렬 여부 |
| `user_id` | uuid/string | 선택 | 세션에서 획득 | 관심 섹터 개인화 정렬 시 사용 |
| `source_screen` | enum | 선택 | `home`, `holding_form`, `detail`, `recommendation` | 검색 호출 화면 |

검색 결과 내부 필드:

| 필드 | 타입 | 출처 | 설명 |
|------|------|------|------|
| `stock_code` | string(6) | `company` | 종목 코드 |
| `corp_name` | string | `company` | 종목명 |
| `corp_code` | string | `company` | DART 고유번호, 존재 시 B4/A1 연계 |
| `sector` | string | `company` | 섹터 |
| `current_price` | int/decimal | `stock_price` | 최근 가격 |
| `change_rate` | decimal | `stock_price` 계산 | 전일 대비 등락률 |
| `market_cap` | bigint | `stock_price` 또는 확장 데이터 | 시가총액, 있으면 정렬 보조 |
| `support_status` | enum | 시스템 계산 | `supported`, `limited`, `unsupported` |
| `match_reason` | string | 시스템 계산 | 종목명 일치, 코드 일치, 섹터 일치 등 |

### 4.4 처리 흐름

1. **검색어 입력 수신**
   - UI는 사용자가 입력한 `query`를 수신한다.
   - 앞뒤 공백을 제거하고 연속 공백을 하나로 정규화한다.
   - 입력값이 비어 있으면 검색 요청을 보내지 않는다.

2. **최소 입력 길이 검증**
   - 한글/영문 종목명 검색은 2글자 이상일 때만 실행한다.
   - 숫자 종목코드 검색은 2자리 이상일 때만 실행한다.
   - 1글자 입력은 성능 보호를 위해 검색하지 않고 안내 문구만 표시한다.

3. **검색 타입 판별**
   - `query`가 숫자로만 구성되면 `code` 검색으로 처리한다.
   - 그 외에는 `name` 검색으로 처리한다.
   - `search_type`이 명시되어 있으면 해당 타입을 우선한다.

4. **종목 마스터 조회**
   - `company` 테이블에서 `stock_code`, `corp_name`, `sector`, `corp_code`를 조회한다.
   - 코드 검색은 `stock_code LIKE '<query>%'`를 우선 적용한다.
   - 이름 검색은 `corp_name ILIKE '%<query>%'` 또는 DB별 동등 검색을 사용한다.
   - MVP에서는 별칭 테이블이 없으면 별칭 검색은 제외한다. 향후 `company_alias` 테이블을 추가할 수 있다.

5. **검색 결과 랭킹**
   - 정확히 일치하는 `stock_code`를 최우선으로 정렬한다.
   - 종목명이 검색어로 시작하는 결과를 다음 순위로 둔다.
   - 종목명 부분 일치 결과를 그 다음 순위로 둔다.
   - 로그인 사용자의 관심 섹터와 일치하는 종목은 동일 점수 내에서 우선 노출할 수 있다.
   - 결과 수는 기본 10개, 최대 20개로 제한한다.

6. **시세 데이터 보강**
   - `include_price=true`이면 `stock_price`에서 각 종목의 최근 가격을 조회한다.
   - 최근 가격과 전일 가격이 모두 있으면 등락률을 계산한다.
   - 시세가 없으면 검색 결과는 유지하되 `price_status=missing`으로 표시한다.

7. **지원 범위 표시**
   - MVP 지원 섹터 또는 분석 가능 universe에 포함되면 `support_status=supported`로 표시한다.
   - `company`에는 있지만 재무/시세/RAG 데이터가 부족할 가능성이 있으면 `limited`로 표시한다.
   - 향후 분석 불가 종목은 `unsupported`로 표시하고 B4/A1 이동을 제한할 수 있다.

8. **자동완성 렌더링**
   - UI는 종목명, 종목코드, 섹터, 현재가, 등락률, 지원 상태를 드롭다운에 표시한다.
   - 결과가 10개를 초과하면 상위 결과만 표시하고 Enter 또는 전체 검색 버튼으로 더 볼 수 있게 한다.
   - 키보드 위/아래 이동, Enter 선택, ESC 닫기를 지원한다.

9. **결과 선택 처리**
   - 사용자가 검색 결과를 클릭하면 `stock_code`를 확정한다.
   - `source_screen=holding_form`이면 B2 보유 종목 등록 폼의 `stock_code`에 값을 채운다.
   - `source_screen=home` 또는 `detail`이면 B4 종목 기본 정보 조회 화면으로 이동한다.
   - 선택 이벤트에는 `stock_code`, `corp_name`, `sector`를 포함한다.

10. **검색 로그 저장**
    - 검색 성공/실패 이벤트는 PII 없이 기록한다.
    - 로그에는 `query_length`, `result_count`, `selected_stock_code`, `source_screen` 정도만 저장한다.
    - 검색어 원문 저장은 운영 정책에 따라 제한하며, 저장하더라도 개인정보성 입력은 마스킹한다.

11. **비용/LLM 처리**
    - B3는 DB 검색 기능이므로 LLM을 호출하지 않는다.
    - 자연어 질의에서 종목을 추론하는 기능은 A5 Curator 또는 별도 고급 기능으로 분리한다.
    - `llm_cost_log`에는 B3 관련 비용 로그를 남기지 않는다.

### 4.6 예외 처리

| 예외 상황 | 감지 조건 | 처리 방안 | 사용자 메시지 |
|-----------|-----------|-----------|---------------|
| 검색어 없음 | trim 후 빈 문자열 | 검색 요청 없음, placeholder 유지 | `종목명 또는 종목코드를 입력해 주세요.` |
| 1글자 입력 | 종목명 1글자 또는 코드 1자리 | 검색 차단, 성능 보호 안내 | `2글자 이상 입력하면 검색됩니다.` |
| 검색 결과 0건 | `company` 조회 결과 0건 | 빈 상태 표시, 다른 키워드 안내 | `검색 결과가 없습니다. 다른 키워드를 시도해 보세요.` |
| 종목 마스터 미적재 | `company` 테이블 없음 또는 0건 | 검색 비활성화, 관리자/데이터팀 확인 필요 | `종목 데이터가 준비되지 않았습니다.` |
| 시세 데이터 결손 | `stock_price` 최근 가격 없음 | 가격/등락률 없이 결과 표시 | `일부 종목은 최신 시세가 없습니다.` |
| DB 검색 실패 | DB connection/query error | 검색 실패 메시지, 에러 로그 저장 | `일시적으로 검색할 수 없습니다. 잠시 후 다시 시도해 주세요.` |
| 과도한 검색 요청 | 짧은 시간 내 다수 요청 | debounce/throttle 적용 | 사용자 노출 없음 또는 `잠시 후 다시 입력해 주세요.` |
| 결과 과다 | 결과 수가 limit 초과 | 상위 결과만 표시, 더 구체적 입력 안내 | `검색어를 더 구체적으로 입력하면 결과가 좁혀집니다.` |
| 지원 제한 종목 | `support_status=limited` | 결과는 표시하되 분석 제한 경고 | `이 종목은 일부 분석 데이터가 부족할 수 있습니다.` |
| B2 연결 시 미로그인 | 검색 결과를 보유 종목 등록에 사용하려 하나 세션 없음 | B1 로그인 화면으로 이동 | `보유 종목 등록은 로그인이 필요합니다.` |
| LLM 호출 발생 위험 | B3 처리 중 LLM 라우터 호출 시도 | 호출 차단, 개발 로그에 정책 위반 기록 | 사용자 노출 없음 |

## 6. KPI / 비용 상한 / 운영 기준

| 구분 | 지표 | 목표/임계값 | 비고 |
|------|------|-------------|------|
| 비용 | B3 LLM 호출 수 | 0회 | 월 5만원 상한 보호 |
| 성능 | 자동완성 응답 시간 | 300ms~1초 이내 목표 | DB 정상 기준 |
| 성능 | 결과 수 | 기본 10개 | UI 과밀 방지 |
| 정확성 | 종목코드 정확 일치 노출 | 1순위 | 코드 검색 기준 |
| 정확성 | 종목명 부분 검색 성공률 | 95% 이상 | `company` 적재 품질 의존 |
| 안정성 | 검색 결과 0건 처리 | 100% | 빈 상태 안내 |
| 연계성 | B2/B4 선택 이벤트 전달 | 100% | `stock_code` 필수 |
| 데이터 품질 | `company.stock_code` 중복 | 0건 | unique 필요 |

운영 정책:

- B3는 모든 분석 기능의 입구이므로 검색 실패 메시지를 명확히 제공한다.
- 검색은 LLM이 아니라 `company` 기반 DB 조회로 처리한다.
- `company` 테이블 적재가 늦어지면 B2, B4, A1~A5 연계 기능이 모두 영향을 받는다.
- 1글자 검색은 성능 보호를 위해 실행하지 않는다.
- 결과에는 지원 상태를 표시해 사용자가 분석 가능 여부를 미리 알 수 있게 한다.
- 검색어 로그는 개인정보가 섞일 가능성을 고려해 최소화하거나 마스킹한다.
